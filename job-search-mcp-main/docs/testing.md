# 테스트 전략

## 현재 상태

이 프로젝트에는 자동화된 테스트(유닛 테스트, 통합 테스트)가 없습니다.

## 테스트를 추가하지 않은 이유

### 1. 핵심 로직이 외부 사이트 스크래핑

프로젝트의 핵심 기능은 잡코리아와 사람인의 HTML을 파싱하는 것입니다.
유닛 테스트를 위해 HTML 목업을 만들 수 있지만, 사이트 구조가 변경되면 목업 기반 테스트는 통과하더라도 실제 환경에서는 실패합니다.
즉, **테스트가 통과해도 정상 동작을 보장할 수 없습니다.**

### 2. MCP 서버 레이어가 얇음

`index.ts`의 서버 코드는 MCP SDK 위에 얇은 핸들러를 등록하는 구조입니다.
SDK 자체의 동작은 SDK 측에서 테스트되므로, 이 프로젝트에서 중복 테스트할 필요가 없습니다.

### 3. 순수 함수가 적고 단순함

`formatJobPostings` 같은 순수 함수가 존재하지만, 로직이 단순하여 테스트 대비 효용이 낮습니다.

## 현재 검증 방법

실제 환경에서의 수동 테스트가 가장 신뢰할 수 있는 검증 방법입니다:

```
# search_jobs 테스트
당근마켓 채용공고 검색해줘
사람인에서 네이버 채용공고 찾아줘
잡코리아에서 삼성SDS 채용공고 2페이지 보여줘

# search_jobs_bulk 테스트
네이버, 카카오, 라인플러스, 당근마켓, 토스 채용공고를 한 번에 검색해줘.
```

대량 검색 성능 테스트는 README의 "대량 검색 성능 테스트 (100개사)" 섹션에 있는 프롬프트를 사용합니다.

## 향후 테스트 추가 기준

다음 조건에 해당하면 테스트 도입을 검토합니다:

- 플랫폼이 3개 이상으로 늘어나는 경우
- 파싱 로직에 복잡한 변환(코드 매핑, 날짜 정규화 등)이 추가되는 경우
- 여러 기여자가 참여하여 회귀 방지가 필요한 경우
